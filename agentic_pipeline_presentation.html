01
Decompose
Break the file into agent-consumable atomic units before any generation

A dedicated Extractor Agent reads each legacy file and splits it into isolated logical blocks. No generation happens yet — only structural analysis. This is the most critical discipline: the agent never touches a monolithic file.

VBScript

  • Split at every Sub / Function boundary
  • Each subroutine becomes one isolated work unit
  • COM object calls flagged for registry mapping
  • Type inference hints extracted per unit

Classic ASP

  • Split into two separate layers per file
  • SQL DAL: string-concatenated query blocks
  • UI Layer: Response.Write calls mixed with HTML
  • Each layer migrated independently
02
Context Assembly
Feed the agent a curated, precise context package — no manual file pasting

Before generating a single line of code, the agent receives a structured context package assembled from three sources: the CLAUDE.md rulebook, the migration-patterns.md dictionary, and live schema from our MCP server.

VBScript Context Package

  • CLAUDE.md: enforce sealed classes, async/await, XML docs
  • migration-patterns.md: COM → .NET service equivalents
  • COM object registry mappings (auto-pulled via MCP)
  • Type inference hints for late-bound variables

Classic ASP Context Package

  • CLAUDE.md: Controller/Service/View separation rules
  • Request.Form → ViewModel property map
  • EF Core entity schema (live from MCP server)
  • Raw SQL → typed DbSet query patterns

Why the MCP Server matters

Without the MCP server, every engineer manually pastes schema files into the prompt. The MCP server auto-injects live codebase schema into every context package — ensuring the agent always works with the real, current data model. This eliminates an entire class of hallucination errors.

03
Generation
Structured slash commands — not free-form prompts — drive deterministic output

The agent executes a slash command from our template library. This is the critical difference from ad-hoc prompting — the command encodes the entire target architecture. The agent produces code constrained by the rulebook.

VBScript → /map-vbscript-to-logic 
// Input: VBScript subroutine unit
Sub ProcessOrder(orderId)
  Dim conn, rs
  conn.Execute "UPDATE orders..."
End Sub

// Output: Sealed C# Service class
public sealed class OrderService
{
  public async Task ProcessOrderAsync(
    int orderId)
  {
    await _context.Orders
      .FindAsync(orderId);
  }
}
Classic ASP → /modernize-asp-view 
// Input: Single .asp file
// Output: THREE files produced

// 1. Controller Action
public async Task<IActionResult>
  Index(OrderViewModel model) { }

// 2. Strongly-typed ViewModel
public class OrderViewModel
{
  public int OrderId { get; set; }
}

// 3. Razor .cshtml View
@model OrderViewModel
<h1>@Model.OrderId</h1>

The key discipline: slash commands, not chat

Free-form prompting produces variable results. Slash commands are parameterized pipeline triggers — they encode architecture decisions, enforce output structure, and produce consistent results across every engineer on the team.

04
Validation Gate
Automated referee runs before any human sees the code

Before any human reviews the output, enterprise_ai_validation_script.py runs automatically via static analysis. Every issue is labelled by severity and blocks promotion to review.

Technology Pattern Caught Severity Why It Matters
VBScript Swallowed exceptions (bare catch {}) CRITICAL Silent failures in production
VBScript Untyped variables (object/dynamic) HIGH Runtime errors, no IntelliSense
VBScript Non-sealed service classes MEDIUM Violates architecture contract
Classic ASP SQL string concatenation CRITICAL SQL injection vulnerability
Classic ASP Direct Session[] access HIGH Bypasses auth middleware
Classic ASP Residual Response.Write calls HIGH UI layer leaking into controller
Classic ASP Synchronous controller actions MEDIUM Thread pool exhaustion at scale
05
Recovery
Targeted fix — never regenerate from scratch

When the validator flags a failure, we do not re-run from scratch. The /refactor-logic command is triggered, passing the validator's structured report back to the agent as input. The agent performs a surgical, targeted fix.

VBScript Recovery Pattern

  • Validator report → agent input (structured JSON)
  • Most common: wrap loose procedural logic inside a proper Service class boundary
  • Agent fixes only the flagged unit — no full regeneration
  • Validator re-runs automatically after fix

Classic ASP Recovery Pattern

  • Most common: business logic leaked into the View layer
  • Agent moves logic back into Controller or Service
  • Enforces separation of concerns without touching the DAL
  • Full audit trail: original → flag → fix → pass

The discipline: structured input, not re-prompting

Most teams re-run the whole prompt when the agent fails. We pass the validator's machine-readable report directly back as the recovery input — the agent knows exactly what's wrong, where it is, and what the fix should look like. This cuts recovery time by ~70% and produces deterministic fixes.

CLAUDE.md — Architecture Rulebook 
# Migration Rules — VBScript

## Output Requirements
- All service classes MUST be sealed
- All public methods MUST be async
- No dynamic or object types permitted
- Constructor injection ONLY for dependencies
- XML doc comments on all public methods

## Naming
- Services: PascalCase + "Service" suffix
- Methods: verb + noun (ProcessOrderAsync)
- No abbreviations in public APIs
